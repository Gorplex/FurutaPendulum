1 /** @page pd_setup Power and Data
2  * This page describes how to connect a PolyDAQ to power and communication
3  * circuits.
4  *
5  * @section pds_power Power
6  * The PolyDAQ 2 has two power connections; either may be used:
7  * - A mini-USB connector supplies power and enables communication to a PC.
8  * If communication is not needed, a USB charger can be used to power the
9  * board for data logging. Note that this is @e not a micro-USB as is used
10  * by many cell phones and e-readers but a slightly larger "Mini B" plug.
11  * - A standard "barrel jack" connector allows the PolyDAQ 2 to be supplied
12  * from a battery or a "wall wart" type plug-in power supply. Such an
13  * external power supply must be rated at 5 volts to 12 volts with at least
14  * 250mA of current capacity. Batteries should supply a voltage between
15  * 5 volts and 9 volts. A direct connection to an automotive electrical
16  * circuit should @b not be used because most automotive circuits produce
17  * voltage spikes that can destroy a PolyDAQ. Use a separate battery or
18  * plug-in USB charger for automotive applications.
19  *
20  * @section pds_serial Serial Communications
21  * The mini-USB connector on the PolyDAQ 2 can be used for serial communication
22  * between the PolyDAQ and a desktop or laptop computer. When a PolyDAQ is
23  * connected via USB to a properly configured PC, a USB serial port connection
24  * is automatically made.
25  * - @b Linux computers are configured to communicate with a PolyDAQ out of the
26  * box -- no additional drivers are needed. The USB serial port will be
27  * named @c /dev/ttyUSB0 if no other similar USB serial device is connected.
28  * If other USB serial devices are connected, the PolyDAQ can be named
29  * @c /dev/ttyUSB1, or @c /dev/ttyUSB2, @e etc.
30  * - @b Windows(tm) computers often need to have drivers installed for the
31  * USB serial chip on the PolyDAQ. Windows(tm) often recognizes the USB serial
32  * chip when it's plugged into the USB port and automatically installs the
33  * correct drivers. When it does not, drivers for the FT232RL chip are
34  * available at @c http://www.ftdichip.com. When the drivers are working, the
35  * PolyDAQ's serial port will then be named @c COM3 or @c COM4 or @c COM5 or
36  * something similar, depending on how many virtual serial port devices have
37  * been plugged into the computer before.
38  * - @b Mac(tm) computers do something really cool and shiny but we're not sure
39  * what because we can't afford one. This section will be updated after we've
40  * had a chance to borrow a Mac(tm) from one of the Cool Kids and try it.
41  *
42  * The USB serial port can be accessed from a desktop or laptop computer using
43  * the PolyDAQ 2 GUI (see the \ref pd_py_gui "PolyDAQ GUI" page for details)
44  * or through a dumb terminal emulator program. A "dumb terminal" is a program
45  * which allows the user to type characters which are immediately sent to the
46  * PolyDAQ and to see characters which the PolyDAQ has sent back; the terminal
47  * is called "dumb" because it does little or no processing of the characters
48  * -- it just sends and receives them unthinkingly. There are many dumb terminal
49  * programs available for free:
50  * - @b Linux computers can use PuTTY, GTKterm, Screen, or Minicom.
51  * - @b Windows(tm) computers can use PuTTY; some older Windows computers have
52  * a program called "Hyperterminal" installed by default.
53  * - @b Mac(tm) computers can use PuTTY through MacPorts, but this seems to be a
54  * lot of hassle to set up. One can also use the terminal program @c screen
55  * to talk to a serial port; see
56  * @c http://apple.stackexchange.com/questions/32834/is-there-an-os-x-terminal-program-that-can-access-serial-ports
57  * for more details.
58  * Terminal emulators must be set to the correct communication mode and rate in
59  * order to successfully communicate with the PolyDAQ. The following settings
60  * must be used:
61  * | | |
62  * |:------------|:-------------:|
63  * | Baud rate | 115200 |
64  * | Parity | None |
65  * | Data bits | 8 |
66  * | Stop bits | 1 |
67  * | Flow control | None |
68  *
69  * The PolyDAQ 2 has a simple terminal interface which allows the user to see
70  * status and data through the dumb terminal. Most commands are single letters.
71  * Tables of the commands that return measurement data are given on the \ref
72  * pd_channels "Channel Commands" page.
73  * Example commands are:
74  * - @b 0 through @b 9 - Display the results of A/D conversions from the A/D
75  * converter's cannels 0 through 9. Many of these channels aren't connected to
76  * anything, so the results won't have meaning.
77  * - @b A through @b F - These are interpreted by the PolyDAQ as hexadecimal
78  * numbers from 10 (A) through 15 (F), and the PolyDAQ returns the results of
79  * A/D conversions from those channels.
80  * - @b X, @b Y, and @b Z - The PolyDAQ returns a raw reading from one of the
81  * built-in accelerometer's axes (if an accelerometer is on the board).
82  * - @b O (uppercase letter O) - Turn on oversampling. If on, all readings from
83  * all channels will be taken the given number of times and the average value
84  * returned. The @b O must be followed by a number between 0 and 250, then a
85  * carriage return. Typed characters may not be echoed, so type carefully.
86  * - @b h or @b ? - Show a help screen listing commands that can be used.
87  *
88  * @section pds_sd_card The Micro-SD Card for Data Logging
89  * The PolyDAQ 2 can log data to a Micro-SD card continuously at up to about
90  * 500 samples per second. Data is saved to comma-separated-variable (CSV) files
91  * which can be read easily by spreadsheet programs such as OpenOffice Calc and
92  * MS-Excel(tm) or imported into mathematical programs such as Matlab(tm) and
93  * GNU Octave. Inexpensive 8 GB cards have been used for testing.
94  * Logging is started automatically when a card is detected in the PolyDAQ and
95  * stopped when the card is removed. Data is written to files which are
96  * automatically named in sequence: @c DATA_000.CSV, @c DATA_001.CSV,
97  * @c DATA_002.CSV, and so on. It is not recommended to delete a file during a
98  * project because the PolyDAQ will re-use its file name, and there is no useful
99  * time and date information stored with a file, so it can become difficult to
100  * determine which data was taken into which file when.
101  *
102  * @subsection pds_sd_led The Orange LED
103  * When not logging data, the PolyDAQ's orange SD indicator LED pulses smoothly
104  * on and off as a "heartbeat" indicator to show that the software is working
105  * properly. When data is being logged, the heartbeat signal is turned off, and
106  * the user should see a series of brief flashes from the orange LED. Each flash
107  * indicates data being written to the SD card. Data which is taken after a
108  * flash from the LED is not saved to the card until the next flash, and if the
109  * card is removed from its socket before the light flashes, that data will be
110  * lost. Therefore, the user should wait until seeing the SD indicator LED flash
111  * at least once @e after having taken important data to remove the card or cut
112  * power to the PolyDAQ.
113  *
114  * @subsection pds_sd_rate Data Rates
115  * The rate at which data can be logged is limited by the speed at which it can
116  * be reliably written to the SD card. The larger the number of channels whose
117  * data is being logged, the more slowly each channel must be taken. If only
118  * one channel of data (plus time stamps) is being recorded, around 500 points
119  * per second can be saved. If four channels are being stored, the maximum rate
120  * is about 200 channels per second. These rates allow data to be taken
121  * continuously for hours, and the data files can grow to many megabytes in
122  * size. For large files, using scripts in a mathematical programming language
123  * such as Matlab(tm) or Octave is usually much more efficient than using a
124  * spreadsheet program for analysis. For small files, spreadsheets are very
125  * convenient.
126  *
127  * If one attempts to take data faster than the PolyDAQ can handle it, the
128  * results are not necessarily completely disastrous because the time at which
129  * each set of data is taken is written in the data file. This allows garbage
130  * data to be removed and whatever good data is available to be used. However,
131  * keeping safely within the speed limits of the PolyDAQ makes data analysis
132  * and filtering @e much easier.
133  *
134  * @subsection pds_sd_config The Configuration File
135  * In order for data logging to work, there must be a file on the SD card called
136  * @c polydaq2.cfg containing information about which data is to be collected and
137  * how often. This file must contain at least the following lines:
138  * - A time per data row line. This line begins with an uppercase @c T followed
139  * by a colon, space, and number of milliseconds per data line. To take data
140  * once every 20 milliseconds:
141  * @code
142  * T: 20
143  * @endcode
144  * - One or more channel configuration lines. Each of these lines begins with an
145  * uppercase @c C, then has a channel command (see the @ref pd_channels
146  * "Channel Commands" page) and numbers specifying calibration slope and
147  * offset. Last comes the channel's name in quotes. To measure strain from
148  * the @c S1 channel (whose channel command is "9"), writing the A/D output
149  * without changing its scale or offset, one would put the following line into
150  * @c polydaq2.cfg:
151  * @code
152  * C: 9, 1.0, 0.0, "Strain 1"
153  * @endcode
154  * An example configuration file follows. In this file, the angle potentiometer
155  * and accelerometer data are saved as calibrated data in physical units, while
156  * the strain data is saved as raw A/D bits, not changed into physical units
157  * before saving on the SD card:
158  * @code
159  * # PolyDAQ2 Logger Configuration File
160  * # Header contains information about configuration of PolyDAQ card.
161  * # The number X-Y-Z shows (Thermocouple channels)-(Voltage channels)-(strain channels)
162  *
163  * H: PolyDAQ 2 0-4-4
164  *
165  * # Time per sample set, in milliseconds.
166  * # All configured channels will be measured at this rate
167  *
168  * T: 5
169  *
170  * # Channel configurations. Each channel is configured with the following
171  * # columns, which must be in this order and separated by commas:
172  * #
173  * # C: The tag for a channel configuration line
174  * # Channel command: Command which reads the given channel
175  * # Gain: Raw data is multiplied by this before saving
176  * # Offset: This is added to data after multiplication
177  * # Name: Name of channel, put into column header in file
178  * #
179  * # Channels will be saved in columns in the data file in the order in
180  * # which channel configuration commands are put in the configuration file.
181  * #
182  * # Example:
183  * # C: 9, 1.0, 0.0, "Cow Strain"
184  *
185  * C: 0, 0.2176, 170.0, "Angle Pot (deg)"
186  * C: 9, 1.0, 0.0, "Strain 1 (bits)"
187  * C: X, 0.0000610, 0.0, "Accel X (g)"
188  * C: Y, 0.0000610, 0.0, "Accel Y (g)"
189  * @endcode
